📌 1. Cambio de Nombre y Contexto (Adaptación del Caso 19 a "Sabores Caseros")

Se mantienen todos los requerimientos funcionales y no funcionales del Caso 19, pero adaptados al contexto de una rotisería que vende platos preparados para llevar.

Contexto adaptado:

La rotisería "Sabores Caseros" es un local dedicado a la preparación y venta de platos caseros listos para llevar. El local es pequeño y tiene capacidad limitada. Actualmente enfrenta una disminución en ventas debido a la falta de un sistema de delivery online, competencia con apps de entrega y limitaciones de espacio.

Solución propuesta:

Implementar un sistema de ventas online que permita a los clientes registrarse, realizar pedidos, pagar en línea, recibir boletas digitales y gestionar entregas a domicilio dentro de un radio definido.

🎯 4. Configuración del Tablero Kanban en Taiga (Basado en 05_TaigaTablero.tex)

Columnas del Tablero:

  1. To Do (Por hacer)

  2. In Progress (En progreso) → WIP ≤ 5

  3. Code Review (Revisión de código) → WIP ≤ 4

  4. Testing (Pruebas) → WIP ≤ 5

  5. Done (Terminado)

Políticas de WIP:

  • In Progress: Máximo 5 tarjetas (1 por persona).

  • Code Review: Máximo 4 tarjetas.

  • Testing: Máximo 5 tarjetas.

  • Swarming: Si Testing llega a WIP, los desarrolladores ayudan a estabilizar.

Etiquetas Obligatorias:

  • business, enabler, auth, catalog, order, payment, delivery, admin, devops, bloqueado, DoD-ok

🧭 5. Pasos para Configurar Taiga.io

  1. Crear las épicas (E1 a E6) en la sección "Epics" de Taiga.

  2. Crear user stories dentro de cada épica, con sus labels correspondientes.

  3. Estimar puntos (ej: 3, 5, 8) para cada story.

  4. Configurar el tablero Kanban con las columnas y WIP limits.

  5. Planificar sprints y asignar stories según la prioridad.

  6. Mover stories entre columnas según su estado (To Do → In Progress → Code Review → Testing → Done).

  7. Usar etiquetas para filtrar y visualizar el progreso por módulo.
